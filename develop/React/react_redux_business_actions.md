# React redux 业务代码放置的位置

这里所说的业务代码代表的是在页面进行操作后，由操作所触发的一轮数据变化。

对于使用 redux 进行状态管理时，业务代码可以有很多地方可以放。比如：

- 业务组件：直接写在业务组件中，Action 和 Reducer 都仅仅是数据传递
- Action：业务组件对数据变化不做处理，直接传入 Action 中，然后在 Action 中通过 getState 函数获取 store 中数据并根据业务需要处理
- Reducer：业务组件和 Action 对数据变化不做处理，在 Reducer 中获取 store 数据，并根据业务需要处理
- 自定义 hooks：将业务处理的函数放入自定义 hooks 中，并提供对应的 Action 函数，使用时直接调用该函数，Action 和 Reducer 都不做进一步处理

以上每种方式都有各自的优缺点，下面就对比一下各自的优缺点。

## 业务代码写在业务组件

优点：

- 所有的逻辑都在组件中，一个文件同时包括了显示逻辑和业务处理逻辑，找代码不用去 Action 或 Reducer 中找了。

缺点：

- UI 逻辑和业务处理逻辑都在一个文件中没有分开，代码阅读起来会相对麻烦一些。
- 如果业务组件触发后需要和 store 中其他数据一起计算时，会需要在组件中获取这些数据，而造成因这些额外数据而引起的性能问题。
- 业务复杂的时候，业务组件会变得非常大，难以维护。

## 业务代码写在 Action

优点

- 业务处理逻辑和 UI 逻辑分离，查看代码相对比较容易了
- 由于 store 数据在 Action 中获取，所以不会有性能问题了

缺点

- 为了使用 gesState，必须给 Redux 引入 thunk。
- 每个业务或 store 中每个 field 都需要一个自己的 Action，相似的或仅传递数据的 Action 和 Reducer 会非常多。

## 业务代码写在 Reducer

优点

- 与将业务放入 Action 中的优点相似
- 另外可以不用引入 thunk 也可以获取到 store 数据了

缺点

- 同业务放到 Action 中相似，每个 store 的字段都需要一个 reducer。这样的话，相似的或仅传递数据的 Reducer 会非常多。
- 业务复杂时，需要复用其他 reducer 业务逻辑时比较麻烦。当然也可以通过将业务逻辑提取到函数中来进行复用。

## 业务代码写在自定义 hooks

如果只是将 UI 中的业务代码提出到自定义 hooks，那其实和放在 UI 组件中是没有差别的。

所以这里所说的是将数据获取和触发业务逻辑一起封装到 hooks 中，做出和 useState 相似的功能，使用元组返回其数据及业务处理函数。

优点

- 数据获取和处理的逻辑在同一个地方，并且和其他 UI 逻辑分离开，代码容易查找
- 如果业务逻辑简单，则 store 中同一层的数据可以通过一组 hooks + action + reducer 来处理，省下很多单调且重复的 Action 和 Reducer 代码
- 通过不同策略，可以让 set 函数接收额外数据或设置多个 store 的字段。

缺点

- 如果没有处理好自定义 hooks，性能问题还是会存在。
- 如果每个业务逻辑都很复杂，那该方法的优势就不存在了，反而还会因为大量的自定义 hooks 引起代码难找的问题，不如写到 Action 中。
